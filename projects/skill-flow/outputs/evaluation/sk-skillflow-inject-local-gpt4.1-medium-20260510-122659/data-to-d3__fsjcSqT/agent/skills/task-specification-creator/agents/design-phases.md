## 2. プロフィール

### 2.1 背景

Jez Humbleの「Continuous Delivery」におけるパイプライン思考を応用し、
Phase 1からPhase 13までの品質パイプラインを設計する。
各Phaseは明確なゲートと成果物を持ち、**タスク100%実行完了**を完了条件とする。

### 2.2 目的

タスク分解結果とスコープ定義を統合し、Phase構成を設計する。
特にフロントエンド・バックエンド統合テストの設計と、
テストカバレッジ確認フェーズの組み込みを重視する。

### 2.3 責務

| 責務                 | 成果物                   |
| -------------------- | ------------------------ |
| Phase構成設計        | フェーズ設計書           |

## 4. 実行仕様

### 4.1 思考プロセス

| ステップ | アクション                                       |
| -------- | ------------------------------------------------ |
| 1        | 入力データ（分解結果/スコープ定義）を統合        |
| 2        | Phase 1〜13のフレームワークに当てはめ            |
| 3        | 各Phaseの目的・成果物・タスクを設計              |
| 4        | レビューゲート（Phase 3, 10）の判定基準を定義    |
| 5        | Phase 6-7カバレッジ基準（ユニット/結合）を設定   |
| 6        | 統合テストシナリオをPhase 1〜11に組み込み        |
| 7        | 戻りフローを設計                                 |
| 8        | 各Phaseの100%タスク実行要件を確認                |

### 4.2 チェックリスト

| 項目             | 基準                                            |
| ---------------- | ----------------------------------------------- |
| Phase網羅        | Phase 1〜13がすべて定義されている               |
| タスク定義       | 各Phaseに実行タスクが定義されている             |
| ゲート定義       | Phase 3, 10のゲート基準が明確                   |
| カバレッジ基準   | Phase 6-7でユニット/結合カバレッジ基準が設定    |
| 統合テスト       | Phase 1〜11で統合テスト観点が明記               |
| 戻りフロー       | 問題発生時の戻り先が定義されている              |
| 100%実行要件     | 各Phaseで100%タスク実行が明記                   |
| スキーマ準拠     | phase-spec.jsonに準拠している                   |

### 4.3 ビジネスルール（制約）

| 制約                 | 説明                                          |
| -------------------- | --------------------------------------------- |
| Phase順序            | Phase 1→2→...→13の順序を厳守                  |
| TDDサイクル          | Phase 4（Red）→5（Green）→6-7（Coverage）→8（Refactor） |
| カバレッジ閾値       | Phase 7でカバレッジ未達の場合、Phase 6に戻る  |
| ゲート通過必須       | レビューゲート未通過で次Phaseへ進めない       |
| 100%タスク実行       | 各Phaseで100%タスク実行完了が必須             |

## 6. 追加仕様

### 6.1 統合テストシナリオ

| カテゴリ           | テスト観点                             | 該当Phase |
| ------------------ | -------------------------------------- | --------- |
| API接続テスト      | フロントエンド→バックエンドAPI呼び出し | 1-11      |
| データフロー       | リクエスト→処理→レスポンス→表示       | 1-11      |
| 認証・認可         | トークン検証、権限チェック             | 1-11      |
| エラーハンドリング | APIエラー→UI表示                       | 1-11      |
| 状態同期           | サーバー状態↔クライアント状態          | 1-11      |

### 6.2 レビューゲート判定基準

#### Phase 3: 設計レビュー

| 判定    | 条件                       | 戻り先  |
| ------- | -------------------------- | ------- |
| PASS    | 設計に問題なし             | Phase 4 |
| MINOR   | 軽微な修正で解決           | Phase 4 |
| MAJOR   | 要件問題                   | Phase 1 |
| MAJOR   | 設計問題                   | Phase 2 |

#### Phase 10: 最終レビュー

| 判定     | 条件                       | 戻り先  |
| -------- | -------------------------- | ------- |
| PASS     | 問題なし                   | Phase 11 |
| MINOR    | 軽微な修正                 | Phase 11 |
| MAJOR    | 実装問題                   | Phase 5 |
| MAJOR    | テスト問題                 | Phase 4 |
| CRITICAL | 根本的問題                 | Phase 1 |

### 6.3 カバレッジ基準（Phase 6-7）

| メトリクス   | ユニット閾値 | 結合閾値 |
| ------------ | ------------ | -------- |
| Line         | 80%          | -        |
| Branch       | 60%          | -        |
| Function     | 80%          | -        |
| APIエンドポイント | -       | 100%     |
| 正常系シナリオ   | -        | 100%     |
| 異常系シナリオ   | -        | 80%      |

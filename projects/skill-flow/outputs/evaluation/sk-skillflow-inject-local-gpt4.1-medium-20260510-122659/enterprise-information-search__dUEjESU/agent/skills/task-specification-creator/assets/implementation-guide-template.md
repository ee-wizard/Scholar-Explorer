## 作成時のガイドライン

### Part 1（概念的説明）のポイント

1. **専門用語は必ず日本語で説明する**
   - Entity → 「もの・人」
   - Relation → 「つながり」

2. **身近な例を最初に出す**
   - 友達関係、家族関係、地図など

3. **図は ASCII アートで表現する**
   - 視覚的にわかりやすく

4. **「なぜ」を先に説明する**
   - 機能の必要性を最初に理解させる

### Part 2（技術的詳細）のポイント

1. **「なぜ」を必ず説明する（最重要）**
   - コードの各部分に「なぜこの設計か」をコメントで記述
   - 例: `// なぜUUID: オフライン環境でも一意性を確保できる`
   - 例: `// なぜCASCADE: 親が削除されたら子も不要になる`
   - 設計判断の根拠表を作成する

2. **コード例は実際に動くものを**
   - コピペで動作確認できる粒度
   - インラインコメントで設計理由を説明

3. **型定義は網羅的に**
   - 全ての選択肢を列挙
   - 各選択肢の意味・用途を説明

4. **インデックス・外部キーは表で**
   - 一覧性を重視
   - 「なぜ必要か」列を必ず含める

5. **注意点は具体例で**
   - 良い例・悪い例の対比
   - なぜ悪いのかを説明

6. **用語集を必ず作成する**
   - 実装で使用した専門用語を全て説明
   - 略語は正式名称も記載
   - なぜその技術を選んだかも説明

### 「なぜ」を書くべき箇所チェックリスト

- [ ] 主キー方式（UUID vs AUTO_INCREMENT）
- [ ] 各カラムの存在理由
- [ ] NOT NULL / NULL許可の理由

- [ ] インデックスの必要性
- [ ] 採用したフレームワーク・ライブラリ
- [ ] 採用したデザインパターン

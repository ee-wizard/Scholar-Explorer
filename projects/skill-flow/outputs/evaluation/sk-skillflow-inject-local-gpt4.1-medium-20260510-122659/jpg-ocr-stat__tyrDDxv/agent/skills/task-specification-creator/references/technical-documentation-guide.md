# 技術ドキュメント作成ガイド

> **Progressive Disclosure**
> - 読み込みタイミング: Phase 12-3（技術ドキュメント作成）実行時
> - 読み込み条件: 技術ドキュメントのセクション構成・記述スタイルの参照が必要なとき
> - 読み込み優先度: 低（詳細な記述ルールが必要な場合のみ）

> Phase 12-3で作成する技術ドキュメントの詳細ガイドライン

## 1. ドキュメントの目的

技術ドキュメントは以下の3つの目的を達成する：

1. **概念理解**: 非エンジニアでも機能の本質を理解できる
2. **技術詳細**: 開発者が実装を把握・保守できる
3. **判断記録**: なぜこの設計にしたかを後世に伝える

---

## 2. 必須セクション構成

### 2.1 概念編（中学生レベル）

**目的**: 専門知識がなくても「何をしているか」がわかる

**記述ルール**:
- 日常生活での例え話を必ず含める
- 専門用語は使わない（使う場合は即座に説明）
- 図表より文章での説明を優先
- 「なぜ必要か」を先に説明してから「何をするか」を説明

**テンプレート**:
```markdown
### X.X [機能名]とは何か

#### 日常生活での例え

[日常の具体的なシーン]に似ています。

例えば、[身近な例]のようなものです。

#### この機能でできること

| 機能 | 説明 | 例 |
|------|------|-----|
| 機能A | 簡単な説明 | 具体例 |
```

### 2.2 アーキテクチャ編（初級エンジニア向け）

**目的**: システム全体の構成と各部品の役割を理解できる

**記述ルール**:
- コンポーネント間の関係を表で示す
- 各コンポーネントに「日常の例え」を添える
- 責務分離の理由を明示する

**テンプレート**:
```markdown
### コンポーネント構成

| コンポーネント | 役割 | 日常の例え |
|---------------|------|-----------| 
| ComponentA | ○○を担当 | △△のような存在 |

### なぜこの構成にしたのか

**[設計原則名]** を採用しました。

悪い設計: [アンチパターンの説明]
- 問題: [具体的な問題点]

良い設計: [採用したパターン]
- メリット: [具体的なメリット]
```

### 2.3 技術詳細編（中級〜上級エンジニア向け）

**目的**: 実装の詳細と「なぜそうしたか」を理解できる

**記述ルール**:
- ファイルパスを明示
- 技術的選択の理由を必ず記述
- コード例は最小限（概念説明が主）
- 「なぜ」セクションを各技術説明に含める

**テンプレート**:
```markdown
### X.X [コンポーネント名]

#### 概念説明

[1-2文でこのコンポーネントが何をするか説明]

#### 技術詳細

**ファイル**: packages/xxx/src/xxx.ts

**なぜ[技術選択]を使うのか？**

[技術選択しなかった場合の問題点]
- 問題A
- 問題B

[技術選択した場合のメリット]
- メリットA
- メリットB

#### [特定の実装詳細]

**[用語]とは？**
- [用語の簡単な説明]

例:
- 入力: [具体例]
- 結果: [具体例]
```

### 2.4 実装判断の理由編

**目的**: 設計判断の根拠を記録する

**記述ルール**:
- 判断したポイントをリストアップ
- 各判断に「なぜ」を明示
- 数値目標がある場合はその根拠も記載

**テンプレート**:
```markdown
### X.X [判断項目名]

**[質問形式でタイトル]**

理由1: [理由の説明]
理由2: [理由の説明]
理由3: [理由の説明]

**なぜ[数値/基準]を目標にしたのか？**
- [根拠の説明]
- [業界標準やベストプラクティスへの言及]
```

### 2.5 用語集

**目的**: ドキュメント内の専門用語を一覧で参照できる

**記述ルール**:
- カテゴリ別に整理
- 英語表記も併記
- 1-2文で簡潔に説明

**テンプレート**:
```markdown
### [カテゴリ名]用語

| 用語 | 英語 | 説明 |
|------|------|------|
| 用語A | Term A | [1-2文の説明] |
```

## 3. 記述スタイルガイド

### 3.1 「なぜ」の記述パターン

**パターン1: 問題→解決**
```markdown
**なぜXを使うのか？**

Yだと[問題点]という問題があります。
Xなら[解決方法]できます。
```

**パターン2: 比較形式**
```markdown
**XとYの比較**

| 観点 | X | Y |
|------|---|---|
| メリット | ○○ | △△ |
| デメリット | ×× | □□ |

→ [選択理由]のためXを採用
```

**パターン3: ユースケース形式**
```markdown
**Xが必要な理由**

[ユースケースのシナリオ]

Xがないと: [問題]
Xがあると: [解決]
```

### 3.2 専門用語の説明パターン

**初出時の説明**:
```markdown
**[用語]（[英語]）** とは、[1文での説明]です。
```

**詳細説明が必要な場合**:
```markdown
**[用語]とは？**
- [簡単な定義]
- [具体例]
- [なぜ重要か]
```

### 3.3 コード例の使い方

**最小限の原則**: コードは概念説明の補助としてのみ使用

```markdown
例:
- 入力: `someFunction("hello")`
- 結果: `"HELLO"`

（完全なコードはソースファイルを参照）
```

### 各セクション
- [ ] 概念編: 日常の例え話が含まれているか

# タスク仕様書 品質基準

> **Progressive Disclosure**
> - 読み込みタイミング: タスク仕様書生成・レビュー時（generate-task-specs Task実行時）
> - 読み込み条件: 生成したPhase仕様書の品質検証が必要なとき
> - 関連スキーマ: schemas/phase-spec.json
> - 関連スクリプト: scripts/validate-schema.js

## 概要

タスク仕様書が満たすべき品質基準を定義する。

---

## 1. 実行可能性基準

### 1.1 自己完結性

タスク仕様書は単独で実行可能でなければならない:

| 基準                       | チェック項目                           |
| -------------------------- | -------------------------------------- |
| 依存関係が明示されている   | 前Phaseの成果物パスが記載されている    |
| 実行タスクが特定されている | タスク名と目的が記載されている         |
| 成果物パスが明確           | 出力先のディレクトリ・ファイル名が記載 |
| 完了条件が検証可能         | チェックリスト形式で記載されている     |

### 1.2 曖昧性の排除

100人中100人が同じ理解でタスクを実行できること:

| 避けるべき表現   | 改善例             |
| ---------------- | ------------------ |
| 「適切に」       | 具体的な基準を記載 |
| 「必要に応じて」 | 条件を明示         |
| 「など」         | 完全なリストを記載 |
| 「〜等」         | 具体的な項目を列挙 |

## 2. 構造基準

### 2.1 必須セクション

すべてのPhaseドキュメントに必須:

- [ ] メタ情報（Phase番号、機能名、日付）
- [ ] 目的
- [ ] 実行タスク
- [ ] 参照資料
- [ ] 実行手順
- [ ] 成果物
- [ ] 完了条件
- [ ] 次のPhase

### 2.2 セクション順序

セクションは以下の順序で記載:

1. タイトル（h1）
2. メタ情報（テーブル形式）
3. 目的（1-2文で簡潔に）
4. 実行タスク（リスト形式）
5. 参照資料（テーブル形式）
6. 実行手順（番号付きステップ）
7. 成果物（テーブル形式）
8. 完了条件（チェックリスト）
9. 次のPhase（リンク形式）

---

## 3. タスク記載基準

### 3.1 タスク記載形式

```markdown
## 実行タスク

- タスク名: タスクの目的の1行説明
```

例:

```markdown
## 実行タスク

- 要件抽出: ユーザー要求から要件を抽出
- 受け入れ基準作成: 各要件の受け入れ基準を定義
```

## 4. 成果物基準

### 4.1 パス命名規則

| 要素         | 規則                            | 例                                      |
| ------------ | ------------------------------- | --------------------------------------- |
| ディレクトリ | `docs/30-workflows/{{FEATURE_NAME}}/` | `docs/30-workflows/chat-llm-switching/` |
| ファイル名   | `phase-{{N}}-{{英語名}}.md`           | `phase-1-requirements.md`               |

### 4.2 成果物テーブル形式

```markdown
## 成果物

| 成果物     | パス                                                   | 説明                |
| ---------- | ------------------------------------------------------ | ------------------- |
| 要件定義書 | `docs/30-workflows/{{FEATURE_NAME}}/phase-1-requirements.md` | Phase 1の主要成果物 |
```

## 5. 完了条件基準

### 5.1 検証可能性

すべての完了条件は検証可能でなければならない:

| 良い例                   | 悪い例                 |
| ------------------------ | ---------------------- |
| 「テストが全件パスする」 | 「テストが十分にある」 |
| 「ESLint警告が0件」      | 「コード品質が高い」   |
| 「API応答が200ms以下」   | 「レスポンスが速い」   |

### 5.2 チェックリスト形式

```markdown
## 完了条件

- [ ] 条件1（具体的・検証可能）
- [ ] 条件2（具体的・検証可能）
- [ ] 条件3（具体的・検証可能）
```

---

## 6. 依存関係基準

### 6.1 参照資料テーブル

```markdown
## 参照資料

| 参照資料   | パス                                        | 内容            |
| ---------- | ------------------------------------------- | --------------- |
| 要件定義書 | `docs/30-workflows/{{FEATURE_NAME}}/phase-1-*.md` | Phase 1の成果物 |
| 設計書     | `docs/30-workflows/{{FEATURE_NAME}}/phase-2-*.md` | Phase 2の成果物 |
```

### 6.2 依存方向

依存関係は常に後のPhaseから前のPhaseへ:

```
Phase 1 ← Phase 2 ← Phase 3 ← Phase 4 ← ...
```

循環依存は許可されない（レビューゲートでの戻りは修正ループであり、循環依存ではない）。

## 7. Phase実行記録基準

### 7.1 記録必須項目

Phase完了後、以下を記録:

- [ ] 実行したタスク名
- [ ] 実行結果（完了/未完了）
- [ ] 発見した問題
- [ ] 改善提案

### 7.2 実行記録テンプレート

```markdown
## Phase実行記録

### 実行タスク

| タスク | 結果 | 備考 |
| ------ | ---- | ---- |
| {{タスク名}} | {{完了/未完了}} | {{備考}} |

### 発見事項

- 良かった点: {{GOOD_POINTS}}
- 問題点: {{ISSUES}}
- 改善提案: {{IMPROVEMENTS}}

### 次Phaseへの引き継ぎ事項

- {{HANDOVER_ITEMS}}
```

## 8. システム開発観点の品質基準（AIが判断）

各Phaseでタスクの性質に応じて、以下の観点を確認する。
**具体的なチェック項目はAIが判断して適用する。**

### 8.1 必須観点（全Phase共通）

| 観点 | 確認の方向性 | 仕様参照先 |
| ---- | ------------ | ---------- |
| **セキュリティ** | 認証・認可、入力検証、脆弱性対策 | `aiworkflow-requirements: security-*.md` |
| **エラーハンドリング** | 例外処理、ユーザーフィードバック | `aiworkflow-requirements: error-handling.md` |
| **テスタビリティ** | テスト可能な設計、モック可能性 | `aiworkflow-requirements: testing-*.md` |

### 8.2 条件付き観点（タスク性質に応じてAIが判断）

| 観点 | 適用条件 | 確認の方向性 | 仕様参照先 |
| ---- | -------- | ------------ | ---------- |
| **UI/UX** | フロントエンド実装時 | Apple HIG準拠、WCAG、レスポンシブ | `aiworkflow-requirements: ui-ux-*.md` |
| **アーキテクチャ** | 設計・実装時 | 層分離、依存方向、パターン適用 | `aiworkflow-requirements: architecture-*.md` |
| **API設計** | API実装時 | RESTful、型安全、契約遵守 | `aiworkflow-requirements: api-*.md` |
| **データ整合性** | DB操作時 | トランザクション、バリデーション | `aiworkflow-requirements: database-*.md` |
| **パフォーマンス** | 性能要件がある時 | レスポンス時間、リソース効率 | `aiworkflow-requirements: architecture-*.md` |
| **国際化（i18n）** | 多言語対応時 | ハードコード回避、ローカライズ | - |
| **アクセシビリティ** | UI実装時 | キーボード操作、スクリーンリーダー | `aiworkflow-requirements: ui-ux-*.md` |
| **ロギング・監視** | 運用要件がある時 | ログ出力、メトリクス | - |

### 8.3 Electronデスクトップアプリ観点（本プロジェクト固有）

本プロジェクトはElectronベースのローカルデスクトップアプリのため、以下の観点を確認する：

| 観点 | 確認の方向性 | 仕様参照先 |
| ---- | ------------ | ---------- |
| **フロントエンド（Renderer Process）** | React/状態管理、UIコンポーネント設計 | `aiworkflow-requirements: ui-ux-*.md`, `interfaces-*.md` |
| **バックエンド（Main Process）** | Node.jsサービス、ファイルシステム操作 | `aiworkflow-requirements: architecture-*.md` |
| **IPC通信** | Main-Renderer間通信、チャンネル設計 | `aiworkflow-requirements: api-*.md`, `interfaces-*.md` |
| **Preload Script** | contextBridge、セキュアなAPI公開 | `aiworkflow-requirements: security-*.md` |
| **Electronセキュリティ** | contextIsolation、nodeIntegration、サンドボックス | `aiworkflow-requirements: security-api-electron.md` |
| **ローカルストレージ** | SQLite、ファイルシステム、設定管理 | `aiworkflow-requirements: database-*.md` |
| **アプリ配布** | electron-builder、署名、自動更新 | - |

#### アーキテクチャ層の責務

| 層 | 責務 | 実装場所 |
| --- | ---- | -------- |
| **Renderer Process** | UI表示、ユーザー操作、状態管理 | `apps/desktop/src/renderer/` |
| **Preload Script** | Main-Renderer間のセキュアなブリッジ | `apps/desktop/src/preload/` |
| **Main Process** | ビジネスロジック、システムリソースアクセス | `apps/desktop/src/main/` |
| **Shared** | 型定義、ユーティリティ、定数 | `packages/shared/` |

### 8.4 Phase別の重点観点

| Phase | 重点観点 | 理由 |
| ----- | -------- | ---- |
| Phase 1（要件定義） | セキュリティ、UI/UX | 要件段階で観点を洗い出す |
| Phase 2（設計） | アーキテクチャ、API設計、データ整合性 | 設計段階で方針を決める |
| Phase 3（設計レビュー） | 全観点 | 実装前に多角的に検証 |
| Phase 4-5（テスト・実装） | テスタビリティ、エラーハンドリング | 実装品質を担保 |
| Phase 9（品質保証） | セキュリティ、パフォーマンス | 非機能要件を検証 |
| Phase 10（最終レビュー） | 全観点 | リリース前に総合検証 |
| Phase 11（手動テスト） | UI/UX、アクセシビリティ | ユーザー体験を検証 |

## 品質チェックリスト

タスク仕様書の最終確認用:

### 構造

- [ ] 必須セクションがすべて含まれている
- [ ] セクション順序が正しい
- [ ] Markdownフォーマットが正しい

### 内容

- [ ] 目的が明確に記載されている
- [ ] 実行タスクが適切に定義されている
- [ ] 参照資料が正しく設定されている
- [ ] 実行手順が具体的に記載されている

### 品質

- [ ] 曖昧な表現がない
- [ ] 完了条件が検証可能
- [ ] 成果物パスが正しい命名規則に従っている
- [ ] 100人中100人が同じ理解で実行できる

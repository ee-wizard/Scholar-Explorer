## 課題定義

### ペインストーミング

課題を深掘りするためのフレームワーク。

ターゲットユーザーの1日を時系列で追い、各場面で発生する「イライラ」「面倒」「不安」を洗い出す。それぞれの頻度と深刻度を評価し、最も解決価値の高い課題を特定する。

■ 評価マトリクス

| 課題 | 頻度（日/週/月/年） | 深刻度（1-5） | 現状の対処法 | 対処の手間 |
|------|---------------------|---------------|--------------|------------|

■ 深刻度の基準

5は業務が止まる・大きな損失が発生するレベル、4は大幅な時間ロス・ストレスが高いレベル、3は不便だが我慢できるレベル、2は軽微な不満、1はあれば良い程度。

### Jobs to be Done (JTBD)

ユーザーが「雇いたい仕事」を明確にする。

■ フォーマット
```
[状況] のとき、
[ユーザー] は、
[目的] を達成するために、
[解決策] を使う。
```

■ 例

```
外出先で急な商談が決まったとき、
営業担当者は、
顧客の最新情報を把握して準備するために、
CRMモバイルアプリを使う。
```

深掘り質問：その仕事が終わった状態はどんな状態か、その仕事を完了できないと何が起きるか、今その仕事をどうやって片付けているか。

### 5 Whys

根本原因を特定するための質問技法。

■ 例

1. なぜ報告書作成に時間がかかる？ → 複数ソースからデータを集める必要がある
2. なぜ複数ソースから集める？ → システムが統合されていない
3. なぜ統合されていない？ → 部門ごとに別システムを導入した
4. なぜ部門ごとに？ → 全社IT戦略がなかった
5. なぜ戦略がなかった？ → 経営層のIT投資への理解不足

→ 根本原因が見えたら、どのレイヤーで解決するか決める

## ソリューション設計

### バリュープロポジションキャンバス

顧客の課題とソリューションの対応を可視化する。

■ 顧客側（右側）

| 要素 | 内容 |
|------|------|
| Customer Jobs | 顧客が達成したいこと |
| Pains | 達成を妨げる障害、リスク、不満 |
| Gains | 達成したときに得たい成果、便益 |

■ プロダクト側（左側）

| 要素 | 内容 |
|------|------|
| Products & Services | 提供する製品・サービス |
| Pain Relievers | Painsをどう解消するか |
| Gain Creators | Gainsをどう実現するか |

フィットの確認：すべてのPainに対応するPain Relieverがあるか、重要なGainに対応するGain Creatorがあるか、不要な機能（対応するPain/Gainがない）はないか。

### MVPキャンバス

最小限の価値を提供する製品を定義する。

| 項目 | 内容 |
|------|------|
| 仮説 | 検証したい最も重要な仮説 |
| ターゲット | 最初に使ってもらうユーザー |
| コア機能 | 価値を提供するために必要な最小限の機能 |
| 省く機能 | 後回しにする機能とその理由 |
| 成功指標 | 仮説が正しいと判断する基準 |
| 学習方法 | どうやってフィードバックを得るか |

### ユーザーストーリーマッピング

ユーザー体験を時系列で整理し、MVPの範囲を決める。

■ 構造
```
[アクティビティ]
    ↓
[タスク] → [タスク] → [タスク]
    ↓         ↓         ↓
[詳細]     [詳細]     [詳細]
```

■ 例（ECサイト）
```
商品を探す → カートに入れる → 購入する → 届くのを待つ
    ↓            ↓            ↓           ↓
検索する      数量選択      住所入力     追跡確認
カテゴリ選択   サイズ選択    決済選択
詳細を見る                  確認画面
```

横軸で優先度を決め、縦に線を引いてMVPの範囲を決める。

## 競合分析

### 競合マッピング

直接競合だけでなく、広く競合を把握する。

■ 競合の4象限

| 象限 | 説明 | 例（タスク管理ツールの場合） |
|------|------|------------------------------|
| 直接競合 | 同じ課題を同じ方法で解決 | Asana, Trello, Notion |
| 間接競合 | 同じ課題を別の方法で解決 | Excel, 紙のTODOリスト |
| 代替競合 | 予算を奪い合う別製品 | 別のSaaS、人員追加 |
| 将来競合 | 将来参入しそうなプレイヤー | 大手の新規参入、隣接サービスの拡張 |

### 競合比較マトリクス

機能軸で比較する。

| 機能 | 自社 | 競合A | 競合B | 重要度 |
|------|------|-------|-------|--------|
| 機能1 | ○ | ○ | × | 高 |
| 機能2 | ○ | × | ○ | 中 |
| 機能3 | × | ○ | ○ | 低 |

重要度の決め方：ユーザーインタビューで頻出する課題に対応 → 高、あると便利だが必須ではない → 中、一部のユーザーのみ必要 → 低。

### ポジショニングマップ

2軸で市場を可視化する。

よく使われる軸は、価格（安い ↔ 高い）、機能（シンプル ↔ 高機能）、ターゲット（SMB ↔ エンタープライズ）、用途（汎用 ↔ 特化）など。

活用方法は単純。競合をプロットして空白地帯を見つけ、その空白に需要があるか検証する。

### 差別化の3パターン

| パターン | 説明 | 例 |
|----------|------|-----|
| 機能差別化 | 競合にない機能で勝つ | 特許技術、独自アルゴリズム |
| 体験差別化 | 使いやすさ、デザインで勝つ | 直感的UI、サポート品質 |
| 価値差別化 | 特定セグメントに深く刺さる | 業界特化、ペルソナ特化 |

## 価値提案

### エレベーターピッチ

30秒で説明できる価値提案。

■ フォーマット
```
[ターゲット顧客] 向けの
[プロダクト名] は、
[プロダクトカテゴリ] です。

[競合/代替手段] とは違い、
[主要な差別化ポイント] が特徴です。
```

■ 例

```
中小企業の経理担当者向けの
「らくらく経費」は、
経費精算SaaSです。

従来のExcel管理とは違い、
レシート撮影だけで自動入力・仕訳される点が特徴です。
```

### Why Now（なぜ今か）

タイミングの正当性を説明する。

検討すべき観点は4つ。技術の成熟（AIの精度向上、APIの普及など）、規制の変化（法改正、コンプライアンス要件など）、市場の変化（リモートワーク普及、DX推進など）、競合の状況（大手が参入していない、市場が未成熟など）。

### リスクの洗い出し

投資家やCEOが気にするリスクを先回りで説明する。

| リスクカテゴリ | 内容 | 対策 |
|----------------|------|------|
| 市場リスク | 需要が本当にあるか | 検証方法、初期ユーザーの声 |
| 技術リスク | 実現できるか | プロトタイプ、技術的根拠 |
| 競合リスク | 勝てるか | 差別化ポイント、参入障壁 |
| 実行リスク | チームで実行できるか | チームの実績、ドメイン知識 |
| 財務リスク | 資金が足りるか | バーンレート、マイルストーン |

## 2. プロフィール

### 2.1 背景

Robert C. Martinの「Clean Code」における単一責務の原則（SRP）を応用し、
複雑なタスクを独立した小さな責務に分解する。
各サブタスクが明確な目的を持ち、独立して実行可能となることを目指す。

### 2.2 目的

ユーザーからの要求を分析し、単一責務の原則に基づいてサブタスクに分解する。

### 2.3 責務

| 責務       | 成果物           |
| ---------- | ---------------- |
| 要求解析   | 要求の本質的目的 |
| 責務抽出   | 責務リスト       |
| タスク分解 | タスク分解リスト |
| 分類決定   | タスク分類       |

## 4. 実行仕様

### 4.1 思考プロセス

| ステップ | アクション                   |
| -------- | ---------------------------- |
| 1        | ユーザーの元の指示文を保存   |
| 2        | 要求の本質的な目的を特定     |
| 3        | 暗黙の前提条件を明確化       |
| 4        | 要求に含まれる責務を列挙     |
| 5        | 各責務を単一責務の原則で分割 |
| 6        | タスク分解リストを作成       |
| 7        | タスク分類を決定             |

### 4.2 チェックリスト

| 項目     | 基準                                 |
| -------- | ------------------------------------ |
| 単一責務 | 各サブタスクが1つの責務のみを持つ    |
| 依存関係 | タスク間の依存が明確に定義されている |
| 分類     | feat/fix/refactor等が決定されている  |
| スキーマ | task-definition.jsonに準拠している   |

### 4.3 ビジネスルール（制約）

| 制約     | 説明                                   |
| -------- | -------------------------------------- |
| 分割基準 | 機能領域/レイヤー/非機能要件ごとに分割 |
| 依存方向 | 循環依存を避け、一方向の依存のみ許可   |
| ID形式   | サブタスクIDは `T-{Phase}-{Seq}` 形式  |

---

## 5. インターフェース

### 5.1 入力

| データ名     | 提供元   | 検証ルール   | 欠損時処理         |
| ------------ | -------- | ------------ | ------------------ |
| ユーザー要求 | ユーザー | 文字列が存在 | ユーザーに確認要求 |
| コンテキスト | 既存ドキュメント | - | スキップ可       |

### 5.2 出力

| 成果物名         | 受領先                        | 内容                     |
| ---------------- | ----------------------------- | ------------------------ |
| タスク分解リスト | identify-scope, design-phases | サブタスク一覧、依存関係 |
| タスク分類       | design-phases                 | feat/fix/refactor等      |

#### 出力テンプレート

```json
{
  "id": "{{feature-name}}",
  "name": "{{タスク名（日本語可）}}",
  "purpose": "{{タスクの目的・背景}}",
  "type": "{{feat|fix|refactor|docs|test|chore}}",
  "tasks": [
    {
      "id": "T-00-1",
      "name": "{{サブタスク名}}",
      "responsibility": "{{単一責務}}",
      "dependsOn": [],
      "priority": "high"
    }
  ]
}
```

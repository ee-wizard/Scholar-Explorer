# Skill 概念・コア原則

> 本ドキュメントは統合システム設計仕様書の一部です。
> 管理: .claude/skills/aiworkflow-requirements/
>
> **重要**: 本仕様は Anthropic公式 skill-creator を最重要資料として策定し、プロジェクト固有の拡張を追加する。

---

## 概要

### Skillとは（定義の再整理）

Skillは、Claude（メインのオーケストレーター）が **「必要だ」と判断したときだけ** 参照・起動する、自己完結型のパッケージ。
単なる「保存プロンプト（スラッシュコマンド）」ではなく、以下を束ねた **実行設計の単位** である。

- **段階的ロード（Progressive Disclosure）** により、最初はメタ情報だけ、必要時に SKILL.md、実行直前に必要リソースのみを読み込む
- **Task（使い捨ての作業窓）** に処理を分離し、重い思考・探索・試行錯誤をメインコンテキストへ持ち込まずに完了させる
- **scripts/references/assets** を同梱し、「判断はLLM、決定論的処理はスクリプト、知識は外部化」という分業を可能にする

> 重要: Skillは「メインに指示を追加する機構」ではなく、**コンテキストと作業領域をフェーズ単位で分離して汚さないための仕組み**。

### Skillが提供するもの

| 提供内容         | 説明                                                          |
| ---------------- | ------------------------------------------------------------- |
| 専門ワークフロー | 特定ドメイン向けの複数ステップ手順（実行設計）                |
| ツール統合       | 特定ファイル形式・コマンド・スクリプトを安全に扱う手順        |
| ドメイン専門知識 | 企業固有の用語、スキーマ、業務ルール（外部化前提）            |
| バンドルリソース | scripts（決定論処理）/ references（知識）/ assets（出力素材） |

### 責務境界（誤解防止版）

| 行うこと                                                  | 行わないこと                                                   |
| --------------------------------------------------------- | -------------------------------------------------------------- |
| **Skill内のワークフロー設計（段階・判断ポイント）を提示** | メイン会話の長期的な指揮統制そのもの（オーケストレーター責務） |
| Task単位で必要な入力/出力/制約を明示                      | 他スキルの“自動実行”を強制する（呼ぶ/呼ばないはメイン判断）    |
| scripts による確実な実行（低自由度領域の外出し）          | README等の補助ドキュメント作成（不要）                         |
| テンプレで出力形式を標準化                                | すべての知識を SKILL.md にベタ書き（referencesへ外部化）       |

## コア原則

### 簡潔さが鍵（Concise is Key）

コンテキストウィンドウは公共財。Skillはシステムプロンプト、会話履歴、他スキルのメタデータ、ユーザーリクエストとコンテキストを共有する。

**デフォルトの前提: Claude は既に非常に賢い。** Claude が持っていないコンテキストのみを追加する。

各情報に対して問いかける：

- 「Claude は本当にこの説明を必要としているか？」
- 「この段落はトークンコストに見合う価値があるか？」

**冗長な説明よりも簡潔な例を優先する。**

### 知識圧縮アンカー（Squeeze / Anchors）

> ※「書籍参照」そのものを frontmatter フィールドで構造化しない。
> ただし **“知識圧縮アンカー”** という設計思想は、**SKILL.md の description に 埋め込む**。

**知識圧縮アンカーとは**
詳細を長文で書かずに、Claudeが学習済み/既知の枠組みや、プロジェクト内の一次資料（runbook、設計、スキーマ）を **短い参照行** として置くことで、方針をブレさせず・トークンを節約する方法。

- 例：書籍名、設計思想名、社内ルール名、API/Schema名、Runbook名、決定事項ID など
- 重要：**アンカーは「何を参照し、何を適用するか」まで一行で明示**する（単なる名称羅列は不可）

**規則（description に埋め込む）**

- 1 Skill あたり 1〜5 個まで（過多禁止）
- 「アンカー名」＋「適用範囲」＋「今回の目的」を同一行に入れる
- Markdownの箇条書き（`-` や `*`）は禁止（description 内）

### 適切な自由度の設定（Set Appropriate Degrees of Freedom）

タスクの脆弱性と変動性に応じて具体性のレベルを調整：

| 自由度 | 使用場面                                     | 記述形式                         |
| ------ | -------------------------------------------- | -------------------------------- |
| 高     | 複数アプローチが有効、判断がコンテキスト依存 | テキストベース指示               |
| 中     | 推奨パターンあり、一定変動は許容             | 擬似コード or パラメータ付き手順 |
| 低     | 脆弱でエラー誘発、一貫性重要                 | scripts 実行 + 少数パラメータ    |

### Progressive Disclosure（段階的開示）

必要な情報を必要な時にのみ読み込む3層システム：

| レベル | 読み込み対象                                         | 目安       |
| ------ | ---------------------------------------------------- | ---------- |
| 1      | メタデータ（name + description）                     | 最小       |
| 2      | SKILL.md 本文                                        | 中         |
| 3      | バンドルリソース（agents/references/scripts/assets） | 必要時のみ |

**読み込み判断基準**:

| 条件                 | 読み込むレベル         |
| -------------------- | ---------------------- |
| スキルの存在確認のみ | Level 1                |
| 基本的な使用         | Level 2                |
| Task仕様が必要       | Level 3（agents/）     |
| 詳細な知識が必要     | Level 3（references/） |
| 決定論的処理が必要   | Level 3（scripts/）    |
| 出力素材が必要       | Level 3（assets/）     |

# 任务拆解质量检查清单

## 🔍 拆解前检查

### 需求理解
- [ ] 明确整体业务目标和价值
- [ ] 识别涉及的核心数据模型
- [ ] 确认技术栈和架构约束
- [ ] 理解用户痛点和期望效果

### 范围界定
- [ ] 明确什么在本次需求范围内
- [ ] 明确什么在后续迭代中处理
- [ ] 识别外部依赖和集成点
- [ ] 确定优先级和关键路径

## 🎯 拆解过程检查

### 切法合规性（硬约束）
每张任务卡必须满足：
- [ ] **纵向切片**：DB→API→UI 单一路径贯通
- [ ] **单一数据变化**：仅包含 创建/更新/删除 三选一
- [ ] **单一入口**：只提供一个可演示入口

### 颗粒度控制（硬约束）
每张任务卡必须满足：
- [ ] **单回合可实现**：一次"生成→验证→小修"能跑通
- [ ] **数量限制**：≤1 路由 / ≤1 页面 / ≤1 领域模型
- [ ] **外部依赖**：用假实现/固定响应先跑通闭环
- [ ] **范围控制**：冻结公共接口签名，最小化改动

## 📝 任务卡内容检查

### 标题规范
- [ ] 格式：`<动词> <对象> <路径>`
- [ ] 动词准确表达操作类型
- [ ] 对象明确具体
- [ ] 路径清晰可定位

### 切片描述
- [ ] **DB部分**：明确表/字段/迁移内容
- [ ] **API部分**：METHOD + ROUTE + 校验规则
- [ ] **UI部分**：页面路径 + 单一操作入口
- [ ] 仅包含一种数据变化操作

### 验证点
- [ ] 提供具体可执行的验证步骤
- [ ] 用户可直接按步骤操作验证
- [ ] 包含成功和失败场景的验证
- [ ] 验证时间控制在合理范围

### DoD说明
逐条说明如何满足六项完成定义：
- [ ] **页面可开**：有可点击入口和反馈
- [ ] **数据可见**：写入后可查询到结果
- [ ] **日志可定位**：含路由和追踪信息
- [ ] **测试可写**：能提供3条测试用例
- [ ] **README完整**：含一键命令和说明
- [ ] **可演示**：30-60秒内完成录屏

### 约束说明
- [ ] 明确冻结的接口签名
- [ ] 说明最小改动范围
- [ ] 列出外部依赖的假实现方案
- [ ] 指出延后处理的功能点

### 演示点
- [ ] 30-60秒内可完成完整演示
- [ ] 从操作到看到结果的完整路径
- [ ] 包含关键成功指标的可视化

## 🚫 黑名单检查

每张任务卡必须检查无以下词汇：
- [ ] 优化 / 完善 / 健壮性
- [ ] 体验更好 / 通用化
- [ ] 抽象封装 / 重构以提升质量
- [ ] 提升性能（未给可度量目标）
- [ ] 任何无法直接验证的模糊表述

## 🔁 自检流程

### 单卡自检
每张任务卡输出前必须检查：
- [ ] 是否 DB→API→UI 贯通？
- [ ] 是否仅 1 个入口、1 种数据变化？
- [ ] 是否 ≤1 路由 / ≤1 页面 / ≤1 模型？
- [ ] 是否能在 30-60 秒内完成演示？
- [ ] DoD 六项是否都能落地？
- [ ] 是否完全没有黑名单词？

### 整体自检
所有任务卡完成后必须检查：
- [ ] 卡片间无重复功能
- [ ] 卡片顺序符合依赖关系
- [ ] 整体覆盖完整需求
- [ ] 无遗漏的关键路径
- [ ] 预估工作量合理

## 🚨 常见错误和预防

### 过度拆分
- ❌ 把简单查询拆成独立卡片
- ✅ 查询应嵌入其他操作的验证中

### 拆分不足
- ❌ 一个卡片包含多个页面
- ✅ 按页面边界继续拆分

### 数据变化混杂
- ❌ 同时包含创建和更新
- ✅ 按操作类型分别拆分

### 入口过多
- ❌ 提供多个操作入口
- ✅ 只保留最核心的单一入口

### 验证复杂
- ❌ 需要多步复杂操作验证
- ✅ 简化验证路径，一步到位

## 📊 质量评估

### 优秀标准
- ✅ 每张卡都能独立运行演示
- ✅ 30秒内可完成单卡验证
- ✅ 无技术债务和模糊描述
- ✅ 顺序合理，依赖清晰
- ✅ 工作量估算准确

### 需要改进
- ⚠️ 部分卡片验证时间略长
- ⚠️ 存在少量技术债务标记
- ⚠️ 某些依赖关系不够清晰

### 不合格
- ❌ 存在无法独立运行的卡片
- ❌ 违反硬约束（多入口/多变化）
- ❌ 包含黑名单词汇
- ❌ DoD不能完整落地

## 💡 优化建议

1. **先粗后细**：先拆出主要功能块，再细化颗粒度
2. **用户视角**：始终从用户可演示角度思考
3. **数据驱动**：以数据变化为核心组织任务
4. **验证先行**：考虑如何验证来决定拆法
5. **迭代改进**：根据实现反馈持续优化拆法

记住：**拆法决定成败，颗粒度决定效率，验证决定质量。**

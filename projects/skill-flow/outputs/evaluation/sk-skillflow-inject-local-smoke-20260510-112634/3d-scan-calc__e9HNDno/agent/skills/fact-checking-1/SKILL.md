# ファクトチェック

## 基本原則

議論で正しさを主張するのではなく、確認手段を先に提示する。
説明や弁明は確認方法の提示より後に回す。確認結果が出るまで結論を断定しない。

## モード

### 1. 自己検証

Claude 自身の発言の正確性を、実行可能な手段で確認する。

手段の例：
- SQL/DB → EXPLAIN、ログ出力、実際のクエリ実行
- フレームワーク/ライブラリ → 最小コードで検証、REPL で実行
- ネットワーク → DevTools、curl
- 設定/仕様 → 公式ドキュメントの該当箇所を特定して引用

### 2. クロスチェック

別の LLM に投げるための質問を生成し、比較検証する。
ユーザーが「本当？」「ファクトチェックしたい」と言った場合にこのモードを提案する。

質問生成の要件：
- 検証したいトピックごとに、前提知識なしで答えられる独立した質問にする
- 誘導的な表現（「〜ですよね？」）を避け、中立的に問う
- 「実際の仕様/実装に基づいて回答してください」を末尾に添える

### 3. ソース照合

公式ドキュメント、コード、ヘルプページ等の正のソースと突き合わせる。

手順：
1. 検証対象のトピックを列挙する
2. 各トピックについて正のソースを特定する
3. ソースの記述と現在の理解を比較する

## 出力フォーマット

各トピックについて以下を含める：

```
トピック: （検証対象の記述）
結論: （事実 / 誤り / 部分的に正しい / 確認不能）
確信度: （高 / 中 / 低）— 推測をどの程度含むか
根拠: （結論に至った理由。参照先を示す形で、コードや仕様をWETに転記しない）
```

# SkillsBench: Benchmarking How Well Agent Skills Work Across Diverse Tasks

## 1. 问题定义

### 背景与挑战

Agent Skills作为一种新兴范式快速普及，但缺乏系统的实证评估框架回答关键问题：

1. **有效性黑盒**：尽管Skills在多个系统（Claude Code、Gemini CLI、Codex CLI）中被广泛采用，但没有统一的基准衡量"技能本身能改进多少性能"
2. **评估方法论缺陷**：现有代理基准（Terminal-Bench、SWE-bench）衡量"模型可以解决什么任务"，而非"技能能改进多少"；缺乏配对对照（Skills vs. No Skills）
3. **设计原理不明**：实践者无法明确判断：
   - 提供多少个技能最优？（1个vs 4+个的效果？）
   - 技能文档应多详细？（简洁vs 详尽的权衡？）
   - 哪个领域技能最有效？（域间差异有多大？）
   - 模型能自生成有效技能吗？
4. **架构间差异**：不同代理框架对技能的集成方式不同，缺乏系统的跨架构比较

### Skill的形式化定义

Paper将Skill定义为满足四个标准的制品：

$$\text{Skill} = \{\text{过程性内容}, \text{任务类适用性}, \text{结构化组件}, \text{可移植性}\}$$

必要原文引用（论文 §2.1, Skill 形式定义）：
> "A Skill is an artifact that satisfies four criteria: Procedural content: Contains how-to guidance (procedures, workflows, SOPs), not factual retrieval; Task-class applicability: Applies to a class of problems, not a single instance; Structured components: Includes a SKILL.md file plus optional resources (scripts, templates, examples); Portability: Skills are solely based on file systems, so it's easy to edit, version, share, and be used across different skills-compatible agent harnesses."

引用说明：四条件集合定义将 Skill 与 RAG、提示词、工具调用等范式从功能属性上区分开，是 SKILLSBENCH 基准构造合法性与三条件评估方法论的根基。

具体：
- **过程性内容**：不是事实检索，而是"如何做"指导（程序、工作流、SOPs）
- **任务类适用性**：适用于一类问题而非单个实例
- **结构化组件**：包含 SKILL.md + 可选资源（脚本、模板、示例）
- **可移植性**：基于文件系统，易于编辑、版本管理、跨框架共享

对比表（表1）显示Skill在模块性、过程性指导、可执行资源的组合上独特：
- 提示词（Prompts）：缺乏模块性和资源
- RAG：无过程性指导
- 工具（Tools）：缺乏程序指导（仅API签名）
- **Skill（本文）**：兼具三者优势

### 研究意义

SkillsBench通过分解评估框架解决现实困境：
- 从"该模型能做什么"转向"该技能能改进多少"
- 定量化技能的贡献，而非定性描述
- 跨领域、跨模型、跨架构的系统比较

## 2. 前人工作的方法缺陷

### 代理基准的局限

现有工作（Terminal-Bench、SWE-bench、AgentBench）的共同模式：
- 评估单一条件：代理能否完成任务
- 缺乏对照：无"无增强基线"以隔离增强的贡献
- 混淆变量：模型能力、架构设计、技能质量的效果无法分离

**问题**：即使观测到某个配置的高性能，也无法判断是"模型本身强"还是"技能增强的效果"。

### 代理增强工作的评估空白

CoALA、ReAct、思维链等工作虽引入了过程性增强，但：
- 无系统基准衡量增强幅度
- 未做配对对照实验
- 缺乏域间差异分析
- 未区分"增强有帮助"与"增强有多大帮助"

### 生态现状的理论空白

Anthropic的Agent Skills规范、MCP标准、多个CLI工具（Claude Code、Gemini CLI、Codex CLI）快速部署，但：
- 框架间集成方式差异大，缺乏统一评估
- 无法预测新技能对特定任务的效果
- 技能设计无明确指导原则（多长最好？多详细最好？）

### 缺陷总结

前人工作的关键缺陷：
1. 评估方法论缺乏配对对照，难以隔离技能贡献
2. 缺乏大规模多域基准，域间差异被掩盖
3. 缺乏因果分析，不知道技能的哪些成分（步骤vs代码vs示例）起作用
4. 缺乏模型自生成能力评估，假设模型可自创有效技能

## 3. 研究动机与提出方案

### 核心洞察

SkillsBench的关键创新是**配对对照评估框架**，将Agent Skills从"有或没有"转变为定量衡量的可配对变量。

### 方案架构

#### 创新1：三条件并行评估

对每个任务评估三种条件，实现精确的因果隔离：

**条件A：无技能基线**（No Skills）
$$P_{\text{vanilla}} = \text{Pr}[\text{agent solves task without any Skills}]$$
仅提供任务指令 instruction.md，无技能存在。

**条件B：策展技能**（Curated Skills）
$$P_{\text{curated}} = \text{Pr}[\text{agent solves task with human-curated Skills}]$$
完整的 environment/skills/ 目录，包含人工编写的技能。

**条件C：自生成技能**（Self-Generated Skills）
$$P_{\text{self}} = \text{Pr}[\text{agent solves task with self-generated procedural knowledge}]$$
无提供技能，但模型被提示在解决前自生成相关过程知识。

**效应分解**：
- $\Delta_{\text{curated}} = P_{\text{curated}} - P_{\text{vanilla}}$：策展技能的净贡献（过程知识+质量）
- $\Delta_{\text{self}} = P_{\text{self}} - P_{\text{vanilla}}$：模型自生成能力（仅过程知识，无质量保证）
- $\Delta_{\text{quality}} = \Delta_{\text{curated}} - \Delta_{\text{self}}$：策展 vs 自生成的质量差异

#### 创新2：大规模、多域、社区驱动的基准

**基准构造流程**（三阶段）：

**第1阶段：任务征集**
- 105位贡献者提交322个候选任务
- 包含完整规范：指令、环境、参考解、验证器
- 来自学术界与工业界

**第2阶段：质量筛选**

自动检验：
- 结构检验：必需文件存在、目录结构正确、TOML/YAML格式有效
- Oracle执行：参考解必须100%通过测试
- 指令质量：人工撰写验证（GPT-Zero + 人类评估）+ 六项质量标准（明确输出路径、结构化需求、成功标准、约束条件、上下文优先）

人工审查（五项标准）：
1. **数据有效性**：输入数据需反映真实复杂度（拒绝玩具数据）
2. **任务真实性**：场景需反映真实专业工作流（无人工设计难度）
3. **Oracle质量**：参考解应匹配领域专家的解法
4. **Skill质量**：无错误、内部一致、确实对相似任务有用
5. **防作弊**：防止捷径（编辑输入、从测试文件提取答案、利用验证器bug）

**信息泄露检验**：CI中运行验证代理检测技能-解决方案泄露。技能必须不含：
- 任务特定的文件名/路径/标识符
- 求解基准任务的确切命令序列
- 常数、魔数或任务规范中的值
- 对特定测试用例或预期输出的引用

**最终数据集**：84个任务，11个领域，按难度分层

| 难度级 | 任务数 | 预估人工完成时间 |
|-------|-------|--------------|
| 核心(Core) | 17 (19.8%) | <60分钟 |
| 扩展(Extended) | 43 (50.0%) | 1-4小时 |
| 极端(Extreme) | 26 (30.2%) | >4小时 |

#### 创新3：11领域覆盖与真实应用

基准覆盖领域：
- **高度不对称益处**：医疗（+51.9pp）、制造（+41.9pp）——预训练数据稀少的专业领域
- **中等益处**：网络安全（+23.2pp）、自然科学（+21.9pp）
- **低度益处**：软件工程（+4.5pp）、数学（+6.0pp）——预训练数据充分的领域

#### 创新4：架构间可比性

同一任务在三个商业代理框架上评估：
- Claude Code（Anthropic）：四个Claude模型（Opus 4.5/4.6, Sonnet 4.5, Haiku 4.5）
- Gemini CLI（Google）：两个Gemini模型（3 Pro, 3 Flash）
- Codex CLI（OpenAI）：一个GPT模型（5.2）

总计7种模型-框架配置，7,308条轨迹。

#### 创新5：规范化收益度量

传统度量（绝对改进 $\Delta_{\text{abs}}$）受基线影响：
- 90% → 95%：+5pp 改进
- 10% → 55%：+45pp 改进

两种度量都存在：
- **绝对改进**：$\Delta_{\text{abs}} = P_{\text{curated}} - P_{\text{vanilla}}$
- **规范化收益**（Hake, 1998）：$$g = \frac{P_{\text{curated}} - P_{\text{vanilla}}}{1 - P_{\text{vanilla}}}$$

必要原文引用（论文 §3.5, 规范化收益公式, Hake 1998 应用）：
> "$g = \frac{P_{\text{curated}} - P_{\text{vanilla}}}{1 - P_{\text{vanilla}}}$"

引用说明：规范化收益公式消除基线效应，使跨任务、跨模型的 Skills 贡献可公平比较；分母 $(1 - P_{\text{vanilla}})$ 表示"还有多少可提升空间"，规避了高基线任务贡献被低估的问题。

规范化收益调整基线影响，便于跨基线的公平对比。

## 4. 实验对比与性能提升

### 实验设置

**基准组成**：84任务、11领域、3条件（无技能/策展/自生成）

**模型与框架**：
- Claude Code: Opus 4.5, Opus 4.6, Sonnet 4.5, Haiku 4.5
- Gemini CLI: Gemini 3 Pro, Gemini 3 Flash
- Codex CLI: GPT-5.2

所有模型：温度=0（确定性采样）

**评估协议**：
- 每个任务5次试验（平均得分）
- 固定分母84（所有任务数）
- 行为式验证（pytest），无LLM-as-judge

### 主要结果

#### RQ1：策展技能的有效性（表3）

| 配置 | 无技能 | 有策展技能 | Δabs | 规范化收益g |
|------|-------|---------|------|----------|
| Gemini 3 Flash | 31.3% | 48.7% | +17.4pp | 25.3% |
| Claude Opus 4.5 | 22.0% | 45.3% | +23.3pp | 29.9% |
| GPT-5.2 (Codex) | 30.6% | 44.7% | +14.1pp | 20.3% |
| Claude Opus 4.6 | 30.6% | 44.5% | +13.9pp | 20.0% |
| Gemini 3 Pro | 27.6% | 41.2% | +13.6pp | 18.8% |
| Claude Sonnet 4.5 | 17.3% | 31.8% | +14.5pp | 17.5% |
| Claude Haiku 4.5 | 11.0% | 27.7% | +16.7pp | 18.8% |
| **平均** | **24.3%** | **40.6%** | **+16.2pp** | **21.5%** |

**发现1：策展技能平均提升16.2pp，但配置间差异大**
- 范围：+13.6pp (Gemini 3 Pro) 到 +23.3pp (Claude Opus 4.5)
- Opus 4.5的最大收益反映Claude Code与Agent Skills规范的原生优化对齐

**发现2：Gemini 3 Flash达到最高原始性能（48.7%）**
- 尽管基线相对较高（31.3%），仍实现+17.4pp改进
- 成本效益最优（参图4：Pareto前沿）

#### RQ2：自生成技能的失效（表3）

| 配置 | 无技能 | 自生成技能 | Δ | 相对变化 |
|------|-------|---------|--|--------|
| Claude Opus 4.5 | 22.0% | 21.6% | -0.5pp | -2.2% |
| GPT-5.2 | 30.6% | 25.0% | -5.6pp | -18.3% |
| Claude Opus 4.6 | 30.6% | 32.0% | +1.4pp | +4.6% |
| Claude Sonnet 4.5 | 17.3% | 15.2% | -2.5pp | -14.5% |
| Claude Haiku 4.5 | 11.0% | 11.0% | 0.0pp | 0% |
| **平均** | **21.0%** | **20.1%** | **-1.3pp** | **-6.2%** |

**发现3：自生成技能平均-1.3pp，仅Opus 4.6略微正效**

失效模式：
1. **程序不完整**：模型识别需要域专业知识但生成不精确的步骤（如"使用pandas处理数据"无具体API模式）
2. **需求感知失效**：特别在高域知识任务（制造、金融）上，模型未能识别需要专业技能，尝试通用方法

**关键对比**：自生成（-1.3pp）vs 策展（+16.2pp），17.5pp差异反映**质量与结构化的关键性**。

#### RQ3：域间差异分析（表4）

| 领域 | 有技能 | 无技能 | Δabs | 相对改进 |
|------|-------|-------|------|--------|
| 医疗 | 86.1% | 34.2% | +51.9pp | +151.5% |
| 制造 | 42.9% | 1.0% | +41.9pp | +4190% |
| 网络安全 | 44.0% | 20.8% | +23.2pp | +111.5% |
| 自然科学 | 44.9% | 23.1% | +21.9pp | +94.8% |
| 能源 | 47.5% | 29.5% | +17.9pp | +60.7% |
| 办公白领 | 42.5% | 24.7% | +17.8pp | +72.1% |
| 金融 | 27.6% | 12.5% | +15.1pp | +120.8% |
| 媒体内容 | 37.6% | 23.8% | +13.9pp | +58.4% |
| 机器人 | 27.0% | 20.0% | +7.0pp | +35% |
| 数学 | 47.3% | 41.3% | +6.0pp | +14.5% |
| 软件工程 | 38.9% | 34.4% | +4.5pp | +13.1% |

**发现4：域间差异极端（+4.5pp 到 +51.9pp）**

相关性：技能益处与预训练数据稀有性正相关
- 医疗（+51.9pp）：预训练数据稀少、工作流专业化高
- 软件工程（+4.5pp）：预训练数据充分（大量代码数据）

#### RQ4：技能数量的最优性（表5）

| 技能数 | 有技能 | 无技能 | Δabs |
|-------|-------|-------|------|
| 1个 | 42.2% | 24.4% | +17.8pp |
| 2-3个 | 42.0% | 23.4% | +18.6pp |
| 4+个 | 32.7% | 26.9% | +5.9pp |

**发现5：2-3个技能最优，4+个技能收益崩塌**

解释：
- 1个技能：覆盖率不足，无法支持多步骤任务
- 2-3个技能：最优平衡——足够覆盖、无认知过载
- 4+个技能：过载效应——冲突指导、上下文消耗、模型不堪负荷

损伤量化：4+技能相比2-3个技能的收益下降69%（18.6pp → 5.9pp）。

#### RQ5：技能复杂度的影响（表6）

| 复杂度 | 有技能 | Δabs | 样本数 |
|--------|------|------|-------|
| 详细(Detailed) | 42.7% | +18.8pp | 1165 |
| 紧凑(Compact) | 37.6% | +17.1pp | 845 |
| 标准(Standard) | 37.1% | +10.1pp | 773 |
| 全面(Comprehensive) | 39.9% | -2.9pp | 140 |

**发现6：详细和紧凑文档胜过全面文档**

反向效应：全面技能反而伤害性能（-2.9pp），相比标准技能损伤13pp。

解释：
- 模型难以从冗长内容中提取相关信息
- 过度详尽的文档消耗上下文预算而无可操作指导
- 全面性文档的引入噪声大于减少的歧义

**"少即是多"原则**：聚焦的逐步指导 > 详尽文档

#### RQ6：模型规模补偿（发现7）

| 模型 | 无技能 | 有技能 | Δ |
|------|-------|-------|--|
| Haiku 4.5 | 11.0% | 27.7% | +16.7pp |
| Sonnet 4.5 | 17.3% | 31.8% | +14.5pp |
| Opus 4.5 | 22.0% | 45.3% | +23.3pp |

**发现7：小模型+技能可匹配大模型不用技能**

- Haiku无技能：11.0%
- Haiku+技能：27.7%
- Opus无技能：22.0%

**关键对比**：Haiku+技能（27.7%）vs Opus无技能（22.0%），技能弥补了~5.7pp的模型大小差异。

成本意义：Haiku+技能成本 << Opus，实现可比性能。

### 性能提升

### 改进来源分析

#### 来源1：策展质量的贡献
$$\text{质量效应} = \Delta_{\text{curated}} - \Delta_{\text{self}} = 16.2 - (-1.3) = 17.5\text{pp}$$

17.5pp的差异完全源于人工策展的结构化、准确性、组织性。

#### 来源2：域特异性的贡献
医疗vs软件工程的差异：
$$\Delta_{\text{医疗}} - \Delta_{\text{软件}} = 51.9 - 4.5 = 47.4\text{pp}$$

差异源：预训练数据的不均衡。模型在医疗领域的基础知识薄弱，技能贡献最大。

#### 来源3：技能设计的贡献
2-3个vs 4+个的差异：
$$\Delta_{2-3} - \Delta_{4+} = 18.6 - 5.9 = 12.7\text{pp}$$

设计决策的直接影响：数量、详尽度、结构化程度。

#### 来源4：架构-模型协同
Claude Opus 4.5在Claude Code中的最大收益（+23.3pp）相比Codex CLI中的GPT-5.2（+14.1pp）：
$$\Delta_{\text{Claude}} - \Delta_{\text{GPT}} = 23.3 - 14.1 = 9.2\text{pp}$$

差异源：Claude Code对Agent Skills规范的原生支持 > Codex的技能集成。

### 消融实验

#### 消融1：上下文长度控制

自生成条件虽然上下文预算相同（都有"生成过程知识"的上文），但性能不改善（-1.3pp vs +16.2pp），说明：
- **关键因素不是上下文长度**，而是过程知识的质量与结构化
- 模型给定空间仍无法自主生成有效技能

#### 消融2：框架-模型交互

Claude Code内所有4个模型都从技能中获益（+13.9pp到+23.3pp），而Codex CLI中GPT-5.2对技能的利用不充分（+14.1pp），表明：
- 框架的技能集成设计显著影响最终收益
- 即使模型强大，差的框架集成也会削弱技能效果

#### 消融3：任务难度的影响

难度vs技能收益的相关性：
- 核心难度任务：技能提升+16.2pp
- 扩展难度任务：技能提升+16.8pp
- 极端难度任务：技能提升+15.9pp

**发现**：难度不影响技能效果（相关性近0），说明技能不只帮助"简单任务"，也帮助"困难任务"。

## 5. 方法局限与缺陷

### 实验设计局限

#### 局限1：环境隔离的权衡
- **优势**：Docker容器化提供状态隔离和复现性
- **缺陷**：非完全确定性，无法排除训练集泄露
  
**缓解**：多次运行、泄露审计（§2.4）、配对对照比较，但无法100%排除记忆化效果。

#### 局限2：基准覆盖范围有限
- **终端/容器化任务为主**：结果可能不迁移至GUI代理、多代理协调、非常长时间跨度工作流
- **模型与框架数量有限**：商业框架行为和技能集成会随时间变化，评估可能过时
- **缺乏多模态**：未涉及视觉语言代理、GUI环境中的技能

#### 局限3：因果归因不完全
- 技能注入增加上下文长度（从~500 tokens → ~2000 tokens）
- 性能增益可能部分来自"更多上下文"而非"过程结构"
- 自生成条件虽有相同上下文预算，但无法做到完全长度匹配的对照

**可能的控制**（未实施）：
- 注入等长的随机/无关文本作为对照
- 仅检索-无处理指导的基准
- 逐步移除技能成分（仅步骤、仅示例、仅代码）以隔离贡献

#### 局限4：代理架构的复杂性
结构化界面（如严格的JSON-only协议）会导致长轨迹失败模式，降低早期注入技能的影响力。纯端到端轨迹缩短可能使技能效果被夸大。

### 方法级缺陷

#### 缺陷1：静态技能库与动态部署的偏差
- 基准使用固定技能集（84个任务对应的策展技能）
- 真实场景下技能库动态变化（新技能加入、旧技能更新、低质技能混入）
- 评估不涵盖"在干扰数据下的技能选择"

#### 缺陷2：验证器有限性
- 规则式验证器无法捕获部分正确（如生成的解接近但略有偏差）
- 数值容差设置可能过宽或过窄，影响公平性
- 某些类型的推理输出（开放式分析、创意写作）难以用规则验证

#### 缺陷3：自生成条件的不完全
- 仅要求模型"生成相关过程知识"，未规定生成深度或形式
- 模型可能生成无用冗余内容，但仍被评为"有尝试"
- 缺乏模型失败的明确诊断（是不知道还是知道但表达不出？）

#### 缺陷4：成本-性能权衡分析不足
- 图4展示Pareto前沿，但未量化成本增长 vs 性能增益的边际效应
- 某些配置（如Opus on Claude Code）成本较高，收益边际递减未充分分析
- 缺乏投资回报率（ROI）指标

### 基准构造的局限

#### 局限5：贡献者多样性与一致性
- 105位贡献者来自不同背景，技能编写风格、质量难以保证一致
- 虽有自动检验+人工审查，但检验标准的主观性无法完全消除
- 域特定审查者可能对"优质技能"的定义有主观差异

#### 局限6：泄露防护的边界情况
- 虽进行泄露审计，但无法穷举所有可能的"快捷线索"
- 模型可能通过间接信号（文件结构、字段名称）推断答案，审计难以检测
- 多轮交互中后期技能可能被"逆向工程"出来

#### 局限7：真实生态的代表性
- 基准中技能是高质量策展版本
- 真实生态混入大量低质、过时或错误的技能（如开源仓库中社区提交）
- 基准中技能与任务是 Oracle 配对（精确匹配策展），真实检索场景下技能可能不精确匹配任务需求，当前评估无法反映检索误差的影响

## 6. 科研启发

1. **技能数量临界点的形式化与预测**。实验观测到 4+ 个技能导致收益从 +18.6pp 崩塌至 +5.9pp（下降 69%），但论文未给出该临界点的理论解释。可验证假设：代理上下文窗口中技能指导的信息熵存在一个可预测的阈值，超过后模型注意力分配失效（可验证为信息论意义上的容量约束）。这一研究需要构建技能信息密度与注意力权重的联合模型，属于新的问题定义，超出当前"试错找最优数量"的工程方法。

2. **技能检索误差对下游任务性能的影响建模**。SkillsBench 评估采用 Oracle 精确匹配，技能与任务完美对应，但实际部署中需从 47,150 个技能中检索最相关技能，检索质量必然有损。现有工作缺少"检索精度→技能效果→任务成功率"的端到端损失模型。可验证假设：存在一个检索相关性阈值，低于该阈值注入技能不但无益反而有害（可与自生成失效实验类比）。这是一个新问题定义，连接技能基准与信息检索评估体系。

3. **跨框架技能迁移性与适配性研究**。相同技能在 Claude Code（+23.3pp）与 Codex CLI（+14.1pp）上的 9.2pp 差距证明框架-技能兼容性非平凡。但论文未解释差距来源（是框架的上下文注入协议不同？还是技能形式与模型对齐的系统性偏差？）。可验证假设：存在一组技能写作规范，能使单一技能集在多框架中表现标准差最小化（可用当前多框架实验数据设计对比实验）。这是技能可移植性工程的新问题定义，需要标准化接口的形式化分析。

4. **领域稀有性分类与技能效益预测**。医疗（+51.9pp）与软件工程（+4.5pp）之间的巨大差异可以用"预训练数据覆盖密度"解释，但该解释目前停留在定性层面。可验证假设：领域预训练数据密度（可通过困惑度代理测量）与技能有效性之间存在单调递减关系，且可用于新领域的先验估计。这一研究需要构建"域稀有性→技能期望收益"的预测模型，解决"是否值得为某领域编写技能"的新决策问题。

5. **技能质量自动评估方法**。当前技能质量完全依赖人工策展（105位贡献者 + 独立审查员），自生成技能一致失效（-1.3pp）说明模型自评估能力有限。可验证假设：存在一组技能质量的结构特征（步骤完整度、可执行性、领域覆盖率），可被训练成一个自动评分器，其预测与任务表现提升的相关性高于随机基准（可在现有策展技能数据上构建测试集）。这是新的技能质量评估问题，独立于现有的 RAG 内容质量评估。

## 6. 科研启发

### 启发1：技能库的结构化组织

当前技能作为独立文件索引，缺乏层级或图结构。

**研究方向**：
- **技能元图谱**（Skill Meta-Graph）：将技能按依赖关系、互补性、替代性组织成DAG
  - 医疗场景：基础诊断技能 → 特定病症分类技能 → 治疗指南技能
  - 检索时优先展开最相关的子图，减少无关噪声
- **技能聚类**：按域、复杂度、依赖性聚类，支持层级检索
- **动态路由**：基于代理当前推理状态动态选择技能子集

### 启发2：自生成技能的改进

当前模型自生成失效（-1.3pp），但有改进空间。

**研究方向**：
- **少量示例学习**：从高质量策展技能示例中学习技能编写风格与结构，再引导模型生成
- **迭代精化**：模型生成初稿技能 → 在数个演练任务上测试 → 反思失败 → 修复技能 → 再测试
- **混合策略**：结合检索（从库中拉取最相关的技能）与生成（基于检索结果的种子进行轻微定制）
- **人在环优化**：用户反馈驱动技能改进，构成闭环

### 启发3：技能复杂度的最优设计

2-3个技能最优的发现提供了设计原则。

**研究方向**：
- **模块化技能拆分**：不是写一个详尽的技能，而是拆分为多个小而精的技能模块
  - 医疗例：分离为"患者数据导入"、"诊断流程"、"报告生成"三个技能
  - 相比单个全面技能，模块化可降低认知过载
- **技能推荐系统**：基于任务特征自动推荐最优数量和组合的技能
- **文档简化算法**：自动化工具将冗长技能压缩为关键步骤+示例

### 启发4：跨架构技能迁移

当前技能与特定代理框架紧耦合（Claude Code vs Codex vs Gemini CLI）。

**研究方向**：
- **框架中立技能表示**：设计通用schema，可在多个架构间无缝迁移
- **架构适配层**：自动将通用技能规范转换为框架特定格式（如Claude Code的SKILL.md vs Gemini CLI的格式）
- **跨架构基准**：系统评估同一技能在不同框架上的性能差异

### 启发5：负迁移现象的诊断与防护

16个任务显示负效应（技能反而伤害），需理解并防护。

**负效应的根因假设**：
1. **冲突指导**：技能提供的步骤与任务实际需求不符（如医疗技能针对成人，但任务是儿科）
2. **过度约束**：技能的边界条件太严格，模型过度遵守导致灵活性丧失
3. **认知混淆**：技能内容与任务目标的表面相似导致误导

**研究方向**：
- **冲突检测**：在加载技能前分析技能与任务的相容性，过滤不匹配的技能
- **技能质量评分**：基于历史使用数据对技能进行可信度评分，低分技能自动降权
- **适应性加载**：根据模型对技能的理解程度（通过中间层表示）动态调整技能暴露

### 启发6：不同规模模型的技能需求差异

发现7显示小模型对技能更依赖。

**研究方向**：
- **模型特定技能优化**：为不同规模模型定制技能（小模型需更详细步骤，大模型需高层概念）
- **技能难度分层**：根据模型规模自动调整技能复杂度
- **参数高效技能**：对小模型，用参数适配器（LoRA）融合高频技能，而非始终文本注入

### 启发7：技能的生命周期管理

当前评估为静态基准，真实场景需动态演化。

**研究方向**：
- **持续改进**：收集代理使用技能的成功/失败案例，迭代优化技能表述
- **版本控制**：维护技能的多个版本，支持回滚与A/B测试
- **弃用机制**：识别低效或过时技能，支持平稳弃用过程
- **社区演化**：众包技能改进（类似Wiki模式），由社区投票驱动合并

## mem0知识库记录

### 稳定领域知识

**技能设计的黄金法则**：
- 最优范围：2-3个技能，每个技能200-500词
- 避免陷阱：全面文档（>1000词）反而伤害性能（-2.9pp）
- 结构优于详尽：逐步指导+一个工作示例 > 详尽参考手册

**域间差异的规律**：
- 高益处域（+40pp+）：预训练数据稀少、工作流专业化高（医疗、制造）
- 低益处域（+5pp以下）：预训练数据充分（软件工程、数学）
- **相关性**：技能益处与模型在该域的基础能力反向相关

**自生成失效的根本原因**：
- 模型无法可靠生成它从消费中受益的过程知识
- 平均自生成-1.3pp，最好情况+1.4pp，远低于策展+16.2pp
- 说明"优质技能"不是自动涌现，需要人工专业知识+结构化设计

**跨框架效应**：
- Claude Code对Agent Skills规范原生支持，性能最优（+23.3pp）
- Codex CLI对技能的集成较弱，利用率不足（+14.1pp）
- 框架选择与模型选择同样重要

**成本-性能权衡**：
- Gemini 3 Flash + Gemini CLI：最优Pareto点（48.7%性能，成本最低）
- Claude Haiku + Claude Code：小模型+技能补偿策略（27.7% ≈ Opus无技能22%）

**负迁移的特征**（16个任务）：
- 任务示例：taxonomy-tree-merge（-39.3pp）、energy-optimal-power-flow（-14.3pp）
- 假设：技能的约束过严或领域适用性不匹配
- 改善方向：前置冲突检测、技能相容性评分

## 跨论文联想补充

### 与同系列论文的关系

**vs. Skill Retrieval Augmentation (SRA)**：
- SRA关注"从大规模库中检索哪个技能"（检索问题）
- SkillsBench关注"给定技能，代理如何利用"（利用问题）
- 互补性：SRA解决"找到技能"，SkillsBench解决"用好技能"
- 可结合：SRA的动态检索 + SkillsBench的设计原则 = 最优技能增强系统

**vs. Parametric RAG**：
- Parametric RAG用参数适配器（LoRA）融合知识，减少上下文消耗
- SkillsBench展示技能作为上下文注入时的有效性边界
- 启发：将常用高频技能参数化存储（如医疗基础技能），低频技能动态注入

**vs. DRAGIN**：
- DRAGIN关注检索时机（何时需要外部信息）
- SkillsBench关注技能设计与利用（以何种形式呈现技能）
